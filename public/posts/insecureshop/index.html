This indicates that interaction with the provider may be partially protected for read access but potentially less restricted for write operations, depending on the implementation of the provider.

Manual Code Analysis #

To try and bypass the login, I started from investiogating the launcvher app that is calling the Login Activity. I then manualkly read through the code to understand the login logic and the first thing I note is that we have Owasp M9: Insecure Data Storage due to unintentional data exposure where the app logs the username and pâssword:

Log.d("userName", username);
Log.d("password", password);

I was able to validate that all user input on the login form is indeed logged:

This could lead to attackers being able to view valid usernames and passwords of users of the platform.

Next, I see the function verifyUserNamePassword() implemented in the Util class. From inspecting the code, it becomes clear that the application hardcodes the valid login credentials inside a HashMap, rather than retrieving them from a secure backend or database.

private final HashMap<String, String> getUserCreds() {
    HashMap userCreds = new HashMap();
    userCreds.put("shopuser", "!ns3csh0p");
    return userCreds;
}

This method creates a HashMap containing a single username–password pair, where the username is shopuser and the password is !ns3csh0p. Because these credentials are embedded directly in the application code, they can easily be discovered through reverse engineering.

The verifyUserNamePassword() function then retrieves this map and checks:

  1. whether the username provided by the user exists as a key in the map.
  2. If the username is found, the function retrieves the corresponding stored password and compares it to the password entered by the user.
  3. If the two values match, the function returns true, indicating a successful login; otherwise, it returns false.

Since the credentials are hardcoded in the application, logging in with the username shopuser and password !ns3csh0p successfully authenticates the user. This demonstrates a security weakness, as anyone analyzing the application code can easily extract these credentials and gain unauthorized access.

This behavior relates to the OWASP Mobile Top 10 risk M9: Insecure Data Storage. Hardcoding sensitive information such as login credentials directly inside the application code is considered insecure because attackers can easily extract these values by reverse engineering the APK. According to OWASP, insecure data storage occurs when sensitive data such as passwords or authentication information is stored without proper protection, encryption, or access controls, allowing attackers to retrieve and abuse the information if they gain access to the

In this case, storing the credentials (shopuser and !ns3csh0p) directly in the application code exposes them to anyone analyzing the app, which could lead to unauthorized access and compromise of user accounts.

You can include it in your pentest report like this, following your previous writing style and adding context:

References to Security Standards:

  • CWE: CWE-532 – Insertion of Sensitive Information into Log File Indicates that logging secrets such as passwords or authentication tokens is unsafe.

  • OWASP MASVS: MSTG-STORAGE-3 States that sensitive data, including authentication credentials, must never be written to logs or other insecure storage.

    Hardcoded Credentials and Sensitive Data in SharedPreferences

Impact: Hardcoding credentials combined with logging them exposes the app to unauthorized access and credential theft, violating secure storage principles and best practices for mobile applications.

AndroidManifest Findings

  1. android:debuggable=true Debugging was enabled on the app which makes it easier for reverse engineers to hook a debugger to it. This allows dumping a stack trace and accessing debugging helper classes.

  2. android:allowBackup=true This flag allows anyone to backup your application data via adb. It allows users who have enabled USB debugging to copy application data off of the device.

  3. A few activities have intent filters defined. If an Activity in an Android app has an intent filter defined in the manifest, it is automatically made accessible to other apps. Without an intent filter (and if exported isn’t set to true), the activity can only be started from within the same app.

Credentials broadcast in plaintext On the About page, when the users click the About InsecureShop button, the onSendData() method is called

Intent intent = new Intent("com.insecureshop.action.BROADCAST");
intent.putExtra("username", userName);
intent.putExtra("password", password);

The snippet above grabs the username and password from shared preferences and broadcasts them as a plain, unprotected broadcast. Since no permission is specified on sendBroadcast(), any app on the device can register a receiver for “com.insecureshop.action.BROADCAST” and silently steal the credentials.

WebViews

I identified, WebViewActivity in the AndroidManifest.xml its configured to be BROWSABLE and it has also defined an android.intent.action.VIEW intent which signifies the presence of a deep link. And this is the data section defined in the intent filter of the WebViewActivity:

<data
                    android:scheme="insecureshop"
                    android:host="com.insecureshop"/>

The deep link scheme registered in the AndroidManifest is insecureshop://com.insecureshop, handled by the exported WebViewActivity. To test exploitability, I crafted a deep link targeting the /web path:

<html>
  <body>
    <a href="insecureshop://com.insecureshop/web?url=https://evil.com">Click me</a>
    <script>
      window.location = "https://evil-insecureshopapp.com";
    </script>
  </body>
</html>

Inspecting the decompiled source in jadx, the /web path extracts the url query parameter and passes it directly to webview.loadUrl() without any validation:

data = data4.getQueryParameter("url");
// ...
webview.loadUrl(data);

I confirmed the deep link triggers the activity and loads the attacker-controlled URL by running:

adb shell am start -a android.intent.action.VIEW \
  -d "insecureshop://com.insecureshop/web?url=https://evil.com"

InsecureShop opened and loaded evil.com inside its WebView, confirming that an attacker can load any arbitrary URL without any restrictions.

Then I started a python server to expose the proof of concept webpage above using the command: python3 -m http.server 8081 and then I visited the localhost URL on my android studio emulator: http://10.0.2.2:8081. I could see evil.com got loaded in the webview of InsecureShop.

This vulnerability aligns with OWASP Mobile Top 10 2024 — M4: Insufficient Input/Output Validation. According to OWASP, mobile applications that fail to properly validate and sanitize data from external sources are at risk of being exploited. In this case, the app fails to validate the url parameter extracted from the deep link before passing it to webview.loadUrl(), allowing an attacker to supply an arbitrary URL. OWASP notes that such weaknesses can lead to unauthorized access to sensitive data and manipulation of app functionality — both of which are achievable here when chained with the insecure WebView settings discussed below.

Improper URL sanitization

While reviewing WebViewActivity, I identified a second deep link path /webview which attempts to restrict URLs to those ending with insecureshopapp.com:

if (StringsKt.endsWith$default(queryParameter, "insecureshopapp.com", false, 2, null)) {
    data = data3.getQueryParameter("url");
}

Although the developer attempted to implement host validation, the use of endsWith() is insufficient. The check only verifies that the URL string ends with insecureshopapp.com without validating the full domain structure. This means an attacker can bypass the check by registering a domain such as evil-insecureshopapp.com and crafting the following deep link:

insecureshop://com.insecureshop/webview?url=https://evil-insecureshopapp.com

This passes the endsWith() check while loading a fully attacker-controlled domain in the WebView. The correct fix would be to validate the full hostname using Uri.getHost() and compare it against an explicit whitelist rather than a suffix match:

Uri parsedUrl = Uri.parse(queryParameter);
if (parsedUrl.getHost().equals("insecureshopapp.com")) {
    // safe to load
}

Make sure that any URLs passed to shouldOverrideUrlLoading(), loadUrl(), or evaluateJavascript() is verified to ensure that any JavaScript passed to the WebView only comes from expected domains.

References:

  • OWASP Mobile Top 10 2024 — M4: Insufficient Input/Output Validation
  • CWE-184: Incomplete List of Disallowed Inputs
